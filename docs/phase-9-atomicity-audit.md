# Phase 9 Atomicity Audit

Phase 9 commerce workflows currently use sequential Supabase client writes. The UI and mutations report partial failures, but these workflows should become PostgreSQL RPC/database transactions before production.

## POS Sale Completion

Current sequential steps:

1. Validate customer, products, selected inventory location, stock, payment method, tendered amount, discounts, and tax.
2. Insert `orders`.
3. Insert `order_items`.
4. Insert `payments`.
5. Insert `receipts`.
6. Update `inventory_items`.
7. Insert `inventory_transactions`.

Partial-failure risk:

- An order can exist without all items, payment, receipt, or inventory ledger rows.
- Inventory can be deducted before a ledger write fails.
- A retry may require manual review.

Recommended RPC boundary:

- `complete_pos_sale(input jsonb)` should perform all order, payment, receipt, inventory, and ledger writes atomically.

## Completed Order Cancellation

Current sequential steps:

1. Read order.
2. Validate status transition.
3. Read original POS sale inventory transactions.
4. Check restoration rows do not already exist.
5. Update each `inventory_item`.
6. Insert restoration `inventory_transactions`.
7. Update order status to `cancelled`.
8. Write audit log.

Partial-failure risk:

- Inventory may restore before order status changes.
- Some items may restore while later items fail.
- Audit log may fail after business state changes.

Recommended RPC boundary:

- `cancel_completed_order(order_id uuid, reason text, staff_id uuid)` should restore inventory and cancel the order in one transaction.

## Return Completion

Current sequential steps:

1. Read return.
2. Validate status.
3. Check prior restoration ledger rows.
4. Restore inventory for restockable items.
5. Insert return inventory ledger rows.
6. Insert administrative refund payment or gift card/store credit when applicable.
7. Update return status to `completed`.
8. Write audit log.

Partial-failure risk:

- Inventory or payment/store-credit rows may be written before return completion status fails.
- Store credit can collide with an existing number.
- Exchange and refund remain administrative tracking only.

Recommended RPC boundary:

- `complete_return(return_id uuid, completion jsonb, staff_id uuid)` should handle inventory, refund/store-credit records, status, and audit atomically.

## Exchange Completion

Current sequential steps:

1. Complete the return administratively.
2. Record exchange notes and audit.
3. No replacement order or payment movement is created yet.

Partial-failure risk:

- Exchange tracking can diverge from actual inventory or payment movement.

Recommended RPC boundary:

- Future exchange RPC should create linked replacement order items, payment balance records, inventory movements, and return completion together.

## Shipment Creation

Current sequential steps:

1. Validate order, fulfillment type, item quantities, origin, address, carrier, and packages.
2. Insert `shipments`.
3. Insert `shipment_items`.
4. Insert `shipment_addresses`.
5. Insert `shipment_packages`.
6. Insert shipment event.
7. Write audit log.

Partial-failure risk:

- Shipment can be created without items, address snapshots, packages, event, or audit log.
- Fulfillable quantities can appear consumed by partial shipment records.

Recommended RPC boundary:

- `create_shipment(input jsonb, staff_id uuid)` should create shipment, items, snapshots, packages, event, and audit in one transaction.

## Inventory Transfer

Current sequential steps:

1. Validate source and destination.
2. Update source inventory item.
3. Update or create destination inventory item.
4. Insert transfer-out ledger row.
5. Insert transfer-in ledger row.

Partial-failure risk:

- Source and destination quantities may diverge if one update succeeds and the other fails.
- Ledger rows may not match current balances after partial failure.

Recommended RPC boundary:

- `transfer_inventory(input jsonb, staff_id uuid)` should update both locations and write both ledger rows atomically.

## Number Generation

Current order, receipt, customer, return, and shipment numbers are generated by reading the latest formatted value and retrying on unique collisions in some flows. This is suitable for development but should be replaced with database sequences or RPC-controlled counters before production.

Status: RPC required before production operational use.
